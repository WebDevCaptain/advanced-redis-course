
Benchmarking

Benchmarking is a technique that you'll use to assess the performance of your Redis server. In this movie, I'll introduce the Redis benchmark utility that you can use with the Redis server to assess the performance of various Redis operations. The redis-benchmark utility simulates running commands done by N clients while at the same time sending M total queries. The utility provides a default set of tests that you can run or you can supply your own custom set of tests. The redis-benchmarking utility supports many different configuration options that you can specify via command line flags, and you can simulate running commands of different types with these different configurations. You can configure the number of clients that are working against the same Redis server, the number of threads on which operations are executed, the pipelining of your server, the payload size, and a whole host of other options. An important rule while you're running your benchmark test is to only compare apples and apples. If you plan to compare Redis with some other kind of database, then it's important that you evaluate the functional and technical differences and take into account what they are before running the benchmark test. For example, it makes complete sense to compare different versions of Redis on the same workload, so you run the same set of benchmark tests on different versions of Redis and see which one performs better. Or you can compare the same version of Redis with different configuration options and see how the benchmark tests perform. Remember that Redis is a server and all commands involve network roundtrips, so it's meaningless to compare Redis to embedded data stores. Redis is an in-memory data store with optional persistence, and if you plan to compare Redis with transactional servers such as MySQL or PostgreSQL, this will not be a direct comparison. You may need to activate certain configuration options for Redis before you can compare a Redis database server with a transactional database.



Benchmarking Redis

In this demo, we'll see how you can run benchmark test on Redis. Now, if you have the Redis server installed, you also have the Redis benchmark utility. So if you just go ahead and run Redis-benchmark --help, you'll get help for how exactly this utility should be used. Here, if you scroll to the top, you'll see all of the options that you can specify with the utility. -h for hostname, -p for port, -a for password, and so on. And if you scroll down here to the very bottom of this output, you'll see different options that you can use to run the benchmark utility. A number of different examples are present here. For example, if you just run redis-benchmark, you're running the utility with the default Redis configuration against the IP address, 127.0.0.1. That's localhost port 6379. Now, this is what we are going to do first, I'm just going to run redis-benchmark in order to run all of the default benchmark tests. These tests might take a few minutes to run through. That's because we get a comprehensive distribution of latencies for all different commands in Redis. LPUSH, MGET, GETSET, and so on. I'll scroll to the top and take a look at a sample of the output here. You can see that the first command that was run was a PING command. A hundred thousand PING operations were executed and completed in 1.4 seconds, 50 parallel clients were used, and the payload size was three bytes. And at the very bottom you can see "multi-thread: no," we did not use multi-threading for this benchmark. For each operation you get a percentile distribution of latencies and the summary at the very bottom. Here you can see a throughput summary of the PING command and you get a latency summary in milliseconds. You have the average latency for a PING operation, that's 0.491 on my machine, the 50th percentile is 0.487 on my machine. And as you scroll down, you'll be able to see the numbers for all of the different operations. Here are the numbers for the SET operation. A hundred thousand requests, 50 parallel clients, three bytes payload, and further down below you have the distribution for throughput and latency for the GET operation. All of the basic operations which you might use for benchmarking are present here in this output. You get latency information, throughput information, and a distribution of latencies. If you run redis-benchmark, it runs a comprehensive set of benchmarking tests and gives you all of the output right here on your screen.
