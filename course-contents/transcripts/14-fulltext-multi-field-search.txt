
Indexing and searching multiple fields

If you have multiple fields with text data, you might want to create an index on multiple fields. And that's exactly what I'm going to do now. I use FT.CREATE as before. The name of the index is idx:vg:multi_fields. Again, using the prefix vg_sales: pay a special attention to the SCHEMA where I have specified multiple fields. We want to index the name field and that contains text data. The description field which also contains text data. But in addition, I've specified the weight of the description field as 5.0. And you specify the weight for text attributes, this declares the importance of this attribute when computing result accuracy. The default weight for any field is one, by indicating that the weight of the description field is five, we are giving it more importance. We are indicating it's a more important field. And finally, the platform field is indexed as a TAG. When you indicate that a field has to be indexed as a TAG, that means the field will be indexed a little differently. Tags allow exact match queries, and are used for categories or primary keys. Now that we've created our second search index, let's take a look at the list of indices that we have so far. FT._LIST will give you this information, and you can see that we have our two indices set up here; name and multi_fields. Now, any search that uses the multi_fields index will search multiple fields, the name field, the description field, and the platform field. Here I search for the term "racing" within my multi_field index. And let's take a look at the results here. You can see that there are two results. That's the first line (integer) 2, and these are two games, vg_sales:011 and vg_sales:002. Notice that for the first document, the term "racing" is present in the description "2005 kart racing video game developed by Nintendo." And if you look at the second result again in the description, you'll find the term "racing." Is a kart racing game developed and published by Nintendo for the Wii. If you don't want the entire document in your search result, you can indicate that only certain fields in the document should be returned. Notice that I searched for the term "racing," but I return just three fields: name, year, and description for the documents in the search result. We get the same two documents in the search result as before, but for each document we only have the name, year, and description. This is a way to trim the size of our results. Search results are ordered based on scores assigned to the documents for the query term that we searched for. Now, based on the scores, vg_sales:011 is at the top of the search results, and vg_sales:002 is in the second place. These scores are specific to the term "racing," that's what we searched for. Now, I'm going to search for the term "race." Notice that the term here in the search query is race and I want to return the name, year, and description. Now, if you look at the search results, you can see that the order is different. We have the same two documents present in the result, vg_sales:002 appears first though, and vg_sales:011 appears second. Based on the scores computed by Redis full text search, the vg_sales:002 document has a higher score for the term "race." If you're curious about the scores for the documents that you get in your search results, you can use the "WITHSCORES" keyword. I've performed a search for the term "race," but I want the results with the scores for this particular term. And you can see that vg_sales:002 has a score of 0.2166, which is higher than the score of 0.2 for vg_sales:011. Now, the scores are computed using something known as TFIDF, or Term Frequency Inverse Document Frequency. If you use the "EXPLAINSCORE" keyword, you'll get a little additional insight into how exactly these scores were computed. TFIDF views the words present in your document based on how often they occur in the document and how often they occur in the entire universe of documents that you have indexed. Again, here you can see that vg_sales:002 has a higher score than vg_sales:011. This is for the term race. Now, let's get the scores and an explanation for the scores for the term "racing." So notice that the term is "racing" in the command. That's what we're searching for, and I've included the "WITHSCORES" and "EXPLAINSCORE" keywords. And here in the results you can see that the score for vg_sales:011 is 1.2, which is higher than the score for vg_sales:002, which is 0.341. We always get our results in the descending order of score.


Realtime updates of indexes

Once you've created an index using documents that have a certain prefix, those indexes are updated in real time. Let me prove this to you. Using the multi_fields index, I'm going to perform a search for the term "xbox." When you run the search, you'll find that there's exactly one record in the result. So there's exactly one game that contains the term "xbox." I'm now going to add two more records with the same vg_sales prefix. This is vg_sales:012. This is a game by Nintendo. Go ahead and add this record using exit, and I'm going to add another Redis hash. This record is for the game, "Grand Theft Auto," which is run on the xbox. Now, having added these two records, these records will be indexed automatically. I'm going to perform a search on multi_fields for the term "xbox," and this time two records will be returned. The second search result here is for the game "Grand Theft Auto." This is a Redis hash that we added to Redis after the multi_fields index was created. The index updated itself in real time to include this as a part of its search.
