
Configuring the benchmark utility

In this movie, we'll explore some of the different flags and options that you can use with the Redis benchmarking utility. We've already run the plane redis-benchmark and looked at the output. Now, let's say your Redis server is not running on localhost and is running on a different port. You might want to use the -h flag to specify the IP address for your Redis server, and the -p flag to specify the port on which Redis is running. Now, the output that you get here will be the same as the output for the redis-benchmark that we ran without specifying any options. I leave it to you to look and interpret this output on your own. I must admit that the complete output is rather verbose. Sometimes we just want a quick overview of the different benchmarking tests that were run. You can get the output in a csv format and this output ends up being much more succinct. Specify the --csv file and here is the output in the csv format. Observe the column headers of the csv file. Test request per second, average latency in millisecond, and the last header is max latency in millisecond. You can see that every row corresponds to a different Redis operation that was performed. PING_INLINE, PING_MBULK, INCR, LPOP, RPOP, LRANGE, and so on. And every record gives you a latency distribution for these commands on your server. If even this is too much detail for you and you want a more condensed output, use the -q flag. Redis-benchmark -q will give you a quick overview of the benchmarking results. Observe that for every operation that is performed, you see the number of requests per second. For example, for the INCR operation, we have 62,893.08 requests per second, and we have the 50th percentile of latency. For example, here it's 0.463 millisecond. Now, let's say you want to run benchmarking tests only for a few operations because those are the operations that you are most interested in. The -q flag will produce a condensed output, and the -t flag is what you use to specify what commands you want to run as a part of your benchmarking test. Here, I'll specify the SADD, SPOP, ZADD, and the ZPOPMIN commands. Only these four commands will be executed, and you can see the benchmark results for these commands printed out on your screen. Now, by default, the benchmarking utility runs every command 100,000 times. You can configure this number to be larger or smaller as you please using the -n flag. -n 500,000 specifies that every command in my benchmark test will be run 500,000 times, and I've used the -t flag to only run the SET and GET commands. Let's go ahead and see the complete output. I haven't specified that I want the condensed output here. If you scroll up, notice that 500,000 requests for SET were completed in 7.97 seconds. And if you scroll down and take a look at the GET operation, once again, there were 500,000 requests for GET that were made and completed in about 9.48 seconds. Now, maybe you want to run benchmark tests for just a single command. You can use a Lua script to specify the command that you want to benchmark. Here is a format of the benchmarking command that you would run using a Lua script. redis-benchmark, and then eval, and then the actual script, the number of keys you want to use, and then the key value pairs, all specified on the command line. Here, I'm benchmarking a single SET operation. I call redis-benchmark and then the keyword "eval," and then my Lua script simply says "return redis.call," perform a SET operation using KEYS[1] and ARGV [1]. The number of keys is one, the key is the term "company," and the value is the word "Loonycorn." This will run benchmarking tests on just the single SET call that I have specified. You can see that the Lua script is executed. There were a total of 100,000 requests that were completed in about 2.3 seconds. And here at the very bottom, there is the latency distribution for the script that we just executed. Let's just switch over to the Redis client and confirm that a value has been set for the key "company." So I do GET company, and you can see the result here says "Loonycorn."
