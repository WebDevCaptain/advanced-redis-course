This is where we'll be running our full text search queries and in a third tab I have my current working directory open, advancedredis. Now this file contains a number of Redis commands to set up my full text search data in the form of Redis hashes. This file contains a series of HSET command to store my data in the form of Redis hashes in my Redis database server. Notice the first command, It's HSET vg_sales: 000, that's the first document. And then I have HSET vg_sales: 001, that's the second document. And then I have HSET vg_sales: 002, that's the third document, and so on. Every record in this document contains information about a certain video game. We have the rank of that video game, the name of that video game, the publisher, the genre and a description. There's a lot of text data here that we can index and search using full text search, and there are a total of 18 records. I'm now going to use pipelining to update my Redis server with all of this data. So I run a cat on vgsales.tet and pipe that through to redis-cli --pipe. These 18 records have been successfully stored in my Redis server. I'll now switch over to the redis-cli and run a simple HGETALL command for vg_sales: 000 to confirm that this Redis hash has been successfully stored. And you can see the structure of the records. This is a record for a single video game, the name is "Wii Sports." The description is the long description that you see on screen. Genre is sports, Global_sales of 82.74 billion. Publisher is Nintendo, platform is the Wii. This was released in the year 2006 and its rank is 1. All 18 records that we've stored here have the same basic structure. I'll now use the FT.CREATE command to create a full text search index on one field of this data. Let's break down this command bit by bit. FT.CREATE will create the search index, then I have the name of the index idx: vg: name. The ON HASH PREFIX 1 vg_sales: indicates the prefix for the documents that we want indexed, vg_sales is the prefix. The Number 1 just means that we have just one prefix for our documents, that is, vg_sales: The SCHEMA Name TEXT indicates what field we want the index to be created on. Here, I've specified just one field, the name of the video game, and I want it indexed as text data. Once you hit enter here, the index will be created. Once the index has been created, you can view information about this index by running theFT.INFO command, specify the name of the index, idx: vg: name, and here you get all of the information about the index that we just created. We have the index name, idx: vg: name. Notice the index definition. The key_type is HASH. There is just one prefix for the documents that we are indexing, vg_sales: and the default score is 1. The index has, of course, a whole bunch of other details which are not really relevant as we get started with search. Now, how do you perform search? You use the FT.SEARCH command. Notice the structure here. The command, FT.SEARCH, the name of the index that we are searching idx: vg: name and the search term, which is just "Sports." An important note here, this particular index has only indexed the text in the name field for our records. So all of the results that you see here and you can see that there are a total of three results, integer 3 tells us there are three results.
