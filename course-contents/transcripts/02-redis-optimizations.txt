Before you start optimizing the performance of your Redis server, you may want to identify where the bottlenecks are. Redis recommends the USE method, utilization saturation and errors method to determine bottlenecks in performance. The USE method is a technique that helps you construct a checklist to identify resource bottlenecks. And what could be the possible bottlenecks in the red server? Well, bottlenecks could be for compute, that is, CPU for latency and memory. These are all parts that could affect your Redis server's performance. If by using the USE methods checklist you've identified compute as your bottleneck, you may want to do CPU profiling to optimize your Redis server. Here, you need to make sure that you configure stack tracers to be available on Redis. Redis as well as any dynamically loaded libraries used with Redis require stack tracers to be enabled for proper on CPU analysis. Make sure you compile Redis with the -02 switch. This means that compiler optimizations are enabled on your Redis server. When you run Redis in production, you'll definitely be using the compiler optimizations. For CPU profiling, you should also ensure that debug information is available with your Redis server. You need to use the compile option -G For this, You also need to ensure that the frame pointer register is present, allowing frame pointer based stack walking. This is important in allowing you to trace your compute calls. The Complete Performance Engineering Guide for CPU profiling in Redis is available here at this URL. You have latency issues with your Redis server if you get slow responses from your server. Latency diagnosis and monitoring is an important part of performance improvement. Now it's possible that you may be running a number of slow commands that are blocking the server. Use the Redis Slow Log feature to check whether you have these slow commands running. If you're running Redis using EC2 instances on AWS, make sure you use HVM based modern EC2 instances like m3.medium. Otherwise the fork operation is very, very slow, thus affecting the performance of your Redis server as a whole. In order to make your OS run faster, make sure you disable transparent huge pages from your OS kernel. If you're running Redis on a virtual machine, it's possible that you have an intrinsic latency that has nothing to do with Redis. Redis on a VM runs slower than Redis on bare metal and finally, Redis offers a latency monitor feature. Use this to get a human readable description of latency events for your server. And the last set of optimizations that we look at will be memory optimizations that we can perform on the Redis server. Now since Redis version 2.0, there are many data types that are optimized to use less space up to a certain size. If you have data types such as Hashes, Lists and Sets which are comprised only of integers, they will be encoded in a very memory efficient manner. These encoded representations use up to ten times less memory and five times less memory is the average saving. If you compile Redis with a 32-bit target, this will use a lot less memory per key since the pointers in Redis will be much smaller, but a 32-bit Redis instance will be limited to 4 GB of maximum memory usage. Recent versions of Redis since Redis 2.0 have introduced new bit and byte level operations which allow us to model a number of problems in very little space using these new primitives. Representing your data using Redis hashes allows your data to be encoded in a very small space. Let's say you're representing users, rather than using different keys for name, last name, email, password, and so on, use a single Redis hash with all the required fields, and that will end up saving a lot of space on the Redis server.
