
Pipelining

In this movie, I'll discuss the first feature that we'll explore in this course. Redis pipelining. Redis pipelining serves to optimize round trip times to the red server by batching your commands together. So you send a batch of commands, they are performed on the redis server and you get a single response for a batch. Let's first understand how the redis server works and what exactly the round trip time is. Redis a TCP server, so you're making a TCP connection to Redis in order to run commands on Redis. The TCP server uses the client-server model, meaning you have the Redis server and you can send commands to the Redis server from the Redis client. Commands that you send to the server are request and what you receive from the server are responses. This is the Request-Response protocol. The query that you write on your Redis client will be sent to the server. The client will then read from the socket connecting to the server in a blocking manner and wait for the response from the Redis server. Redis server will of course process the command that you have sent. It will run the command, retrieve the results for the command, and send the response back to the client. The time taken for this sequence of operations is the round trip time. The time taken for the request to go from the client to the server and the response to come back from the server to the client, this is round trip time. Now, by default, the commands that you run on the Redis client are sent one at a time. So you'll basically run a command which does exactly one thing that will be sent to the server. The server will perform that operation and then return a response. Okay, done to the client. By default, you receive a response for each command that you run on the Redis server. And even if the command runs very fast, you can imagine that there is a round trip time associated with each query or command. You can improve the performance of your server using pipelining. This is because your Redis server can process new requests even if the client has not read the old responses that the server has sent down to the client. This in turn implies that you can send multiple commands to the server without waiting for replies at all, so you need not wait for the reply for the first command before sending, say 10 or 20 commands to the server. So you send commands as a batch and once you send commands as a batch, you can then read all of the replies in a single step. With pipelining enabled, you send a batch of commands in one go, so you'll ask the server to do10 different operations or 50 different operations based on the number of operations you want to perform in a batch, and then you'll receive responses for all of these operations in one go as well. Every request response to the Redis server has a round trip time cost associated with it. With pipelining, you pay this RTT cost for a batch of commands rather than for every single command. Pipelining greatly improves the throughput of your operations on Redis. It improves the number of operations you can perform per second.
